# Obsidian-Assistance 项目经验总结（2026-07-01）

> 本文档属于 **Obsidian 知识库辅助项目** 的公开/半公开工程经验总结。  
> 不包含 Obsidian vault 正文、课程讲义、原始素材、ASR/OCR 全文、私人笔记或 `.obsidian` 真实配置。

## 1. 项目定位

`Obsidian-Assistance` 不是知识库本体，而是辅助本地 Obsidian 知识库运转的工具与流程仓库。

它负责沉淀：

- 课程处理流程规范；
- 辅助脚本；
- 可复用 CSS / UI 思路；
- 数据边界规则；
- 本地核验与自检方法；
- Hermes/Agent 可复用技能设计；
- 项目经验与交接文档。

它不负责保存：

- vault 正文；
- 原始课程文件；
- 音视频、PDF、压缩包；
- ASR/OCR 全文；
- 私人路径配置；
- 账号、token、密钥。

## 2. 本阶段完成的核心经验

本阶段围绕本地 Obsidian 知识库完成了一个较长的自动化处理闭环，沉淀出以下可复用经验：

1. **课程入库可以批量自动化，但必须保留可信度边界。**
   - 有讲义/OCR/转写的课程，可以进入较高置信层级。
   - 只有目录或视频标题的课程，只能标记为“标题级高置信”。
   - 未转写视频不能被描述成“逐讲内容已核验”。

2. **技术自检和内容可信度审计是两件事。**
   - JSON 可解析、链接无缺失、乱码清零、Git 干净，只说明 vault 技术状态健康。
   - 课程内容是否准确，还需要本地源核验、公开资料交叉校对，必要时还要视频转写。

3. **全网比校不应替代课程本地源。**
   - 网络资料适合校对公共知识、术语定义、软件功能、行业方法论。
   - 网络资料不应用来伪造课程讲义、老师案例或未转写的视频内容。
   - 对课程独有观点，应保留“课程说法 / 公开资料 / 判断 / 后续动作”的差异记录。

4. **长循环任务需要明确停止协议。**
   - 用户输入“停止/暂停/打断/不要继续”时，必须停在最近安全点。
   - 安全点应满足：写入已完成或未开始、验证已跑、Git 状态可解释。
   - 任务恢复时，先检查磁盘和 Git 状态，不盲信压缩上下文里的 todo。

5. **Obsidian vault 的 UI 改造应使用“使用层”，不要重排存储层。**
   - 课程正文、卡片、索引保留在稳定路径。
   - 用 dashboard、bookmarks、领域主页、课程门户建立用户面对的导航层。
   - 这样既保持链接稳定，又能逐步提升体验。

## 3. 推荐的分层架构

```text
原始资料层       只读：课程文件、视频、PDF、压缩包
工作产物层       可清理：转写、OCR、中间报告、临时核验
正式 vault 层    只保存最终课程门户、卡片、索引、导入报告
辅助仓库层       只保存脚本、流程、技能、经验总结、脱敏文档
```

本仓库属于“辅助仓库层”。

## 4. 课程处理闭环

推荐标准流程：

```text
素材识别
  → 本地转写/OCR
  → 本地源核验
  → 内容过滤
  → 课程门户/总结/工作流
  → 知识卡片/复习卡片/术语索引
  → 导入报告
  → 技术自检
  → 本地 Git 备份
  → 全网比校/可信度报告
```

### 4.1 本地源核验

优先级：

1. 本地讲义 / PDF / OCR；
2. 音频或视频转写；
3. 同一内容多份转写交叉对比；
4. 课程目录 / 文件名 / 视频标题；
5. 公开网络资料。

公开网络资料只作为公共知识校验，不替代本地源。

### 4.2 内容过滤

应进入正式库：

- 老师主线讲解；
- 方法论；
- 操作流程；
- 术语定义；
- 可复习问题；
- 可复用模板。

应过滤或只在报告中说明：

- 学员个人分享；
- 获奖名单；
- 运营通知；
- 平台推广；
- 二维码/联系方式；
- 临时活动；
- 侵犯隐私或版权风险的原文大段摘录。

## 5. 技术自检清单

每轮写入后至少检查：

```text
1. git status --short
2. 核心 JSON 是否可解析
3. Markdown 是否存在乱码标记
4. href 链接是否缺失
5. frontmatter 是否异常
6. 空文件数量
7. 正式课程门户数量与导入报告数量是否一致
8. 是否误提交媒体/中间产物/vault 配置
```

实践中发现，技术自检能及时发现：

- 旧版聚合文件乱码；
- JSON BOM/格式异常；
- 课程聚合索引与正式课程文件不一致；
- 压缩上下文后 todo 与真实 Git 状态不一致。

## 6. 全网比校 / 可信度审计

新增了可复用思路：`obsidian-web-crosscheck`。

### 6.1 可信度等级

| 等级 | 含义 | 推荐处理 |
|---|---|---|
| A | 有本地讲义/转写/OCR，且公开权威资料支持 | 可长期使用 |
| B | 有本地文档/课程结构，公开资料支持公共知识 | 保留并补来源 |
| C | 主要是标题/目录级依据，未转写视频 | 标记待转写确认 |
| D | 来源弱或公共资料无法支撑 | 暂不扩写 |

### 6.2 适合校对的公开资料

- 软件官方文档：Adobe、Figma、OpenAI、Anki 等；
- 设计系统资料：Material Design、NN/g、Figma Learn；
- 心理学教材/机构：OpenStax、APA；
- 算法/AI教材：CLRS、Google ML Crash Course；
- 运营增长资料：AARRR、内容营销、产品增长框架。

### 6.3 报告格式

每门课程建议输出：

```text
93_导入报告/<课程名>_全网比校与可信度报告.md
```

报告包含：

- 本地依据；
- 公开校对来源；
- 模块/术语对照；
- 判断；
- 待转写确认项；
- 是否建议修正文档。

## 7. 长循环经验

### 7.1 安全边界

允许：

- 写入 vault 的最终 Markdown、索引、报告；
- 修改 Obsidian CSS snippet、bookmarks、模板配置；
- 写入辅助仓库中的流程文档和脚本；
- 本地 Git commit。

禁止：

- 删除/移动原始课程资料；
- 解压不明大压缩包；
- 运行 EXE；
- 上传 vault 内容；
- 把原始课程、转写全文、OCR 全文放进 GitHub；
- 擅自推送敏感内容。

### 7.2 上下文压缩恢复

恢复长任务时：

1. 先查 Git 最新提交；
2. 查工作树是否干净；
3. 查目标课程文件是否实际存在；
4. 如果已完成，不重复写入；
5. 只做轻量验证并推进任务状态。

## 8. Windows / Obsidian 实践注意事项

- E 盘写入优先用 Python；Node.js 在某些环境会出现 EPERM。
- Git Bash 环境下使用 POSIX 语法，不使用 PowerShell 内置命令。
- 百度同步盘可能影响 JSON/BOM，要用 `utf-8-sig` 读、`utf-8` 写。
- `.obsidian/plugins/` 下不要堆备份文件，备份放到辅助项目 archive 目录。
- Obsidian 插件报错要优先检查 JSON 结构，比如 cmdr 的 `leftRibbon` 必须是数组。

## 9. 本仓库应继续沉淀的内容

建议后续继续补充：

2. `scripts/web_crosscheck_stub.py`  
   读取课程门户，生成全网比校报告模板。

3. `templates/web-crosscheck-report.md`  
   独立的全网比校报告模板。

4. `docs/course-loop-operations.md`  
   长循环课程入库操作手册。

5. `docs/vault-health-check.md`  
   自检与修复手册。

## 10. 当前阶段总结

本阶段最大的结论：

> Obsidian 知识库自动化不是简单地“批量生成笔记”，而是一个本地证据、内容过滤、结构化入库、技术自检、公开资料校对共同构成的系统工程。

对辅助项目来说，最有价值的不是保存某次处理结果，而是保存：

- 可重复的流程；
- 失败后的修复方法；
- 数据边界；
- 可信度分层；
- 自动化脚本；
- 可迁移的经验。

因此，本仓库应继续保持轻量、脱敏、流程化，只作为本地 Obsidian vault 的辅助工程层存在。
